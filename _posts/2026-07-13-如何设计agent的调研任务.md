主流开源方案已经收敛到同一个形态：编排者 + 并行子研究员，你的「串行慢」正是这个模式要解的问题 - 星数最高的两个项目都是这么做的。gpt-researcher ([https://github.com/assafelovic/gpt-researcher)（实时](https://github.com/assafelovic/gpt-researcher)%EF%BC%88%E5%AE%9E%E6%97%B6) 28K stars）是 planner-executor-publisher 三段式：planner 把问题拆成互相独立的子问题，每个子问题触发一个爬取 agent 并发跑，publisher 最后聚合成报告；Dify Blog ([https://dify.ai/blog/enhancing-gpt-researcher-with-parallel-and-advanced-iterative-features](https://dify.ai/blog/enhancing-gpt-researcher-with-parallel-and-advanced-iterative-features)) 给出的并行化收益是「总耗时 ≈ 串行总时长 ÷ 最大并行度」。关键洞察：并行的单位是「子问题」，不是「工具调用」 - 先拆问题再并发，而不是把一条串行链里的步骤硬改成异步。

LangChain 的 open_deep_research ([https://github.com/langchain-ai/open_deep_research)（实时](https://github.com/langchain-ai/open_deep_research)%EF%BC%88%E5%AE%9E%E6%97%B6) 12K stars）是当前最值得抄的参考实现 - 按 LangChain 官方博客 ([https://blog.langchain.com/open-deep-research/)，它分三个阶段：Scope（先和用户澄清、把对话历史压缩成一份](https://blog.langchain.com/open-deep-research/)%EF%BC%8C%E5%AE%83%E5%88%86%E4%B8%89%E4%B8%AA%E9%98%B6%E6%AE%B5%EF%BC%9AScope%EF%BC%88%E5%85%88%E5%92%8C%E7%94%A8%E6%88%B7%E6%BE%84%E6%B8%85%E3%80%81%E6%8A%8A%E5%AF%B9%E8%AF%9D%E5%8E%86%E5%8F%B2%E5%8E%8B%E7%BC%A9%E6%88%90%E4%B8%80%E4%BB%BD) research brief）→ Research（supervisor 派发并行子研究员）→ Write（一次性写报告）。supervisor 有一个 think_tool 做反思，决定继续派 ConductResearch 还是宣布 ResearchComplete；每个子研究员上下文隔离，各自迭代搜索，返回前先把发现压缩掉无关 token，per Sergey Bolshchikov 的内部机制拆解 ([https://www.bolshchikov.com/p/open-deep-research-internals-a-step)。DeepWiki](https://www.bolshchikov.com/p/open-deep-research-internals-a-step)%E3%80%82DeepWiki) ([https://deepwiki.com/langchain-ai/open_deep_research/3.1-langgraph-implementation](https://deepwiki.com/langchain-ai/open_deep_research/3.1-langgraph-implementation)) 有完整的 LangGraph 工作流图。

Anthropic 的生产级经验直接回应了你「作答环堆约束机制」的顾虑 - Anthropic 工程博客 ([https://www.anthropic.com/engineering/multi-agent-research-system)（编排者-工作者模式，Opus](https://www.anthropic.com/engineering/multi-agent-research-system)%EF%BC%88%E7%BC%96%E6%8E%92%E8%80%85-%E5%B7%A5%E4%BD%9C%E8%80%85%E6%A8%A1%E5%BC%8F%EF%BC%8COpus) 领队 + 并行 Sonnet 子代理，比单 Opus 提升 90.2%）的核心教训不是给作答模型加更多约束，而是把约束前移到委派契约和输入质量：每个子代理拿到明确的 objective、输出格式、工具指引、任务边界；作答环只吃压缩后的带引用发现，写作本身保持简单。ByteByteGo 的复盘 ([https://blog.bytebytego.com/p/how-anthropic-built-a-multi-agent](https://blog.bytebytego.com/p/how-anthropic-built-a-multi-agent)) 补充了他们早期的翻车：简单问题也狂开子代理、重复搜索，后来靠「按问题复杂度分级投入」（简单问题 1 个 agent 3-10 次调用，复杂问题才开多子代理）修掉 - 这对你很重要：并行化要带 effort scaling，不是所有问题都值得开满。

社区讨论本身这 30 天偏冷（19 条里只有 5 条来自最近 7 天），热度在工程博客而非 Reddit - 引擎只在 r/LocalLLaMA ([https://reddit.com/r/LocalLLaMA](https://reddit.com/r/LocalLLaMA)) 和 r/LangChain ([https://reddit.com/r/LangChain](https://reddit.com/r/LangChain)) 捞到 2 条相关帖，HN 上「deep research agent」没有近期大讨论。这个领域的定型知识沉淀在上面几篇文章和两个头部仓库里，方向已经比较稳定，不是每周在变的赛道。

KEY PATTERNS from the research:
1. 拆问题 → 并发子研究员 → 聚合，是三个头部方案（GPT Researcher / open_deep_research / Anthropic）共同的骨架 - per LangChain 官方博客 ([https://blog.langchain.com/open-deep-research/](https://blog.langchain.com/open-deep-research/))
2. 每个子研究员上下文隔离，返回前压缩发现；对话历史先压成 research brief 再进研究环 - per Sergey Bolshchikov ([https://www.bolshchikov.com/p/open-deep-research-internals-a-step](https://www.bolshchikov.com/p/open-deep-research-internals-a-step))
3. 约束放在委派契约（目标/输出格式/工具/边界）而不是作答环的事后管制 - per Anthropic 工程博客 ([https://www.anthropic.com/engineering/multi-agent-research-system](https://www.anthropic.com/engineering/multi-agent-research-system))
4. 按问题复杂度分级投入（effort scaling），防止简单问题被多代理放大成本 - per ByteByteGo ([https://blog.bytebytego.com/p/how-anthropic-built-a-multi-agent](https://blog.bytebytego.com/p/how-anthropic-built-a-multi-agent))
5. 作答/写报告是单独、简单的最后一步，只消费带引用的压缩发现 - per Tavily Docs ([https://docs.tavily.com/examples/open-sources/gpt-researcher](https://docs.tavily.com/examples/open-sources/gpt-researcher)) 那你再综合看一下这里面的内容呢
